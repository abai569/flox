# 002 自动购流与自动续费修复

## 背景

用户反馈自动购流功能异常；同时决定将自动续费链路一并整理修复。排查后确认自动购流主要问题集中在套餐模式的扣库存顺序、不过期用户筛选、前端启用流程和金额单位转换；自动续费主流程可用，但仍存在仓储边界不清和测试覆盖不足的问题。

## 已确认问题

| 问题 | 位置 | 影响 |
|------|------|------|
| 套餐模式先扣库存再扣款加流量 | `go-backend/internal/http/handler/jobs.go` | 余额不足或购买事务失败时库存已减少，造成库存丢失 |
| 自动购流候选排除不过期用户 | `go-backend/internal/store/repo/repository_mutations.go` | `exp_time = 0` 的用户不会自动购流 |
| 仪表盘启用时先提交无套餐配置 | `vite-frontend/src/pages/dashboard.tsx` | 页面显示已启用，但后台候选查询不会扫描到该用户 |
| 管理员自定义购流价格单位不一致 | `vite-frontend/src/pages/user.tsx` | 前端按元输入，后端按分扣款，价格会低 100 倍 |
| 创建/更新用户忽略自动购流配置写入错误 | `go-backend/internal/http/handler/mutations.go` | 配置保存失败时前端仍显示成功 |
| 自动续费任务直接访问 `repo.DB()` | `go-backend/internal/http/handler/jobs.go` | 破坏仓储边界，续费后续状态同步难以保证原子性 |
| 自动续费成功后的恢复动作分散 | `go-backend/internal/http/handler/jobs.go` | 扣款成功后用户流量重置、隧道恢复、日志写入不在统一仓储封装内 |
| 自动续费专项测试不足 | `go-backend/internal/http/handler/jobs_renewal_test.go` | 缺少用户自动续费成功/余额不足/关闭开关等回归保护 |

## 修复目标

1. 自动购流购买行为具备事务一致性。
2. 不过期用户也能按剩余流量阈值触发自动购流。
3. 前端不会产生“显示已启用但配置不可执行”的半启用状态。
4. 价格单位在用户页与后端保持一致。
5. 参数校验失败时明确返回错误，避免静默失败。
6. 自动续费逻辑尽量收口到仓储方法，减少 handler 直接操作数据库。
7. 自动续费补充专项测试，覆盖成功与失败路径。

## 实施方案

### 后端

1. 调整候选用户查询：允许 `exp_time = 0` 或 `exp_time > nowMs`。
2. 新增或重构自动购流仓储方法，将以下操作放入同一事务：
   - 读取用户并校验余额
   - 校验套餐类型、启用状态和自动购流开关
   - 校验并扣减有限库存
   - 扣减余额
   - 增加用户流量配额和 `traffic_flow`
   - 写入 `user_traffic_buy_log`
   - 写入签名 `balance_log`
3. `handleAutoBuyTraffic` 不再先调用独立扣库存方法，改为调用事务化购买方法。
4. 创建/更新用户时校验自动购流参数：
   - 套餐模式要求套餐存在、类型为 `traffic`、启用、允许自动购流
   - 自定义模式要求购买量和价格均大于 0
   - 配置写入失败必须返回错误
5. 余额不足、套餐无效、库存不足保持不中断其他用户处理，并写清晰日志。

### 自动续费整理

1. 将续费成功后的用户重置、`user_tunnel` 同步、状态恢复等动作收口到仓储方法，避免 handler 直接调用 `repo.DB()`。
2. 为自动续费补充明确的仓储封装，至少覆盖：
   - 续费成功后的用户状态恢复
   - 用户与隧道流量归零
   - 到期失败后的禁用与日志写入
3. 保持现有业务行为不变：
   - 已过期且开启自动续费、余额充足时自动续一个月
   - 余额不足或关闭自动续费时按原逻辑停用并归零
4. 为 `TryAutoRenewForUser` 和到期任务补充测试，避免未来回归。

### 前端

1. 仪表盘启用自动购流时先打开设置弹窗，不立即提交开关。
2. 只有保存套餐和阈值成功后才显示自动购流已启用。
3. 取消设置时保留原状态，不生成无套餐配置。
4. 用户页自定义购流价格：
   - 后端返回分，表单展示元
   - 提交时元转换为分
5. 用户页根据购买方式清理无关字段，避免套餐模式和自定义模式字段混用。

## 测试清单

- [ ] 不过期用户 `exp_time = 0` 在剩余流量低于阈值时可自动购流。
- [ ] 套餐模式余额不足不扣库存、不加流量、不写购买日志。
- [ ] 套餐模式购买事务失败不扣库存。
- [ ] 套餐库存不足不扣余额、不加流量。
- [ ] 套餐模式购买成功后同时扣款、加流量、扣库存、写购买日志和余额日志。
- [ ] 自定义模式价格按分扣款，前端展示仍为元。
- [ ] 仪表盘打开开关但取消弹窗时，自动购流状态不被错误启用。
- [ ] 保存套餐配置后仪表盘刷新显示正确套餐和阈值。
- [ ] 自动续费成功后扣款、延长到期、重置流量、恢复用户和隧道状态。
- [ ] 自动续费余额不足时不扣款，用户按原逻辑停用并记录归零历史。
- [ ] 自动续费关闭时到期用户不会误续费。
- [ ] `TryAutoRenewForUser` 在管理员补余额后可恢复过期用户。
- [ ] `go test ./internal/http/handler/...` 通过。
- [ ] `go test ./...` 通过。

## 风险与注意事项

- 套餐库存扣减需要兼容无限库存 `stock = -1`。
- 自动购流定时任务每 10 分钟执行一次，修复后仍可能有最多 10 分钟触发延迟。
- 不应修改 `install.sh` 或 `panel_install.sh`，发布流程会覆盖安装脚本。
- 前端没有测试框架，只做类型检查和构建验证。
- 自动续费修复以“收口实现、补测试、保持现有行为”为主，不引入新的续费周期规则。
